# MEMORY.md

## Long-Term Context

- The user wants realism over convenience: every crafting action should match PoE2DB/编年史 mod pools, tiers, weights, item levels, and currency behavior.
- The simulator should support starting from real semi-finished items, not only white bases.
- The user explicitly expects self-verification after each implementation before final reporting.
- After a fix is verified and intended for the public site, commit and push it to GitHub by default instead of leaving it only in the local workspace.
- Cache and generated data should stay under the project workspace, not the system drive.

## Decisions

- Base item data is loaded from `data/poe2db-base-data.js`, generated from PoE2DB equipment pages cached under `.cache/`.
- Modifier data is loaded from `data/poe2db-mod-data.js`.
- Currency and special crafting factor data is loaded from `data/poe2db-crafting-data.js`.
- `CraftingCore` is the source of truth for item construction, validation, currency behavior, mod eligibility, and rendering.
- UI should call core APIs instead of duplicating crafting validation.
- Custom starting items are created through `CraftingCore.makeCustomItem`, so selected affixes are still validated against the real current base, item level, rarity, slots, and occupied affix groups.

## Current Constraints

- Some generated text may contain legacy mojibake in old labels, but core data and page behavior should remain functional.
- Full equipment coverage creates large audit reports, so `tools/audit-currency-pools.mjs` uses summarized entries with bounded detailed samples.
- Local browser verification is expected when UI changes are made and the local static server can run.

## Recent Corrections

- Affix-cap modifiers must flow through `CraftingCore.capFor()`. This includes base implicits such as Dusk/Gloam/Penumbra/Tenebrous jewellery (`允许的前缀/后缀 +/-#`) and socketed Soul Core `Serle's Triumph / 瑟尔的凯旋` (`允许的后缀 +1`). Smoke now verifies Dusk Ring is 4 prefixes / 2 suffixes and that Serle's Triumph opens a fourth suffix slot that Exalted Orb can fill.
- Deployment zips must store paths with forward slashes, for example `data/poe2db-mod-data.js`. PowerShell `Compress-Archive` on this Windows workspace can produce entries like `data\poe2db-mod-data.js`; some static hosts then fail to serve `/data/*.js`, causing the production data-load warning. Build deployment zip with `System.IO.Compression.ZipArchive` entry names normalized to `/`.
- Frontend usability improvements should preserve dense simulator workflows over decorative layout: keep currency controls reachable, expose usable/unusable state, let users filter custom modifier selects by rendered modifier text/group/tier, and verify changes in the browser against real imported data.
- Rings must not receive boot-only or shield-only mods.
- Desecrated mods start unrevealed and require Abyssal Echoes to reveal.
- Bright Omen should only remove desecrated mods.
- Fractured mods should be visibly highlighted.
- Removal previews for combinations such as reduction + perfect chaos should reflect the actual candidate chosen by tier/level rules.
- Full equipment support introduced special shield cases; pure evasion or energy shield rune shields need compatible shield pools instead of empty pools.
- Desecration material coverage must include jewellery plus talismans for lockbones, armour/offhand classes including shields, bucklers, and foci for ribs, and weapons plus quivers for jawbones. Smoke tests now verify all 29 classes with desecrated modifier data can actually roll a hidden desecrated mod for their own class.
- Desecrated modifier routing must not use an all-equipment fallback for rows that explicitly mention a class. PoE2DB notes that revealed desecrated modifiers may include base modifiers, sceptres have no exclusive desecrated modifiers, and Lich-specific omens only apply to weapons and jewellery. Explicit quiver/shield/crossbow/mace/staff/wand/focus/bow/sceptre rows are narrowed in the importer and rechecked by `tools/audit-desecration-routing.mjs`.
- Abyssal Echoes is modeled as a pending three-choice reveal; the selected choice replaces the hidden desecrated placeholder and is then written to history.
- Sceptre desecration must not require the source text to explicitly say "sceptre"; otherwise broad Desecrated_Modifiers rows are incorrectly removed from sceptre pools.
- Desecration routing regression checks for sceptres, wands, staves, crossbows, quivers, shields, and foci are now part of `tools/smoke-crafting-core.mjs`, so the approved smoke command covers the previous routing audit risk.
- Preserved Cranium / jewel desecration is implemented with a fallback `jewel_abyss` base, imports all 32 PoE2DB `JewelsDesecratedMods` rows, and smoke compares name, level, prefix/suffix, and source text against `.cache/Desecrated_Modifiers.html`.
- Preserved Vertebrae / waystone desecration is implemented with a fallback `waystone_t15` base, imports all 19 PoE2DB `DesecratedWaystoneMods` rows, and smoke compares name, level, prefix/suffix, and source text against `.cache/Desecrated_Modifiers.html`.
- Ordinary equipment desecration must also be smoke-verified row-for-row: `.cache/Desecrated_Modifiers.html` `DesecratedMods /198` is now compared by name, level, prefix/suffix, and source text against imported `sourceSection === "DesecratedMods"` rows.
- The UI desecration pool view should show the item-level/base reference pool with `{ ignoreItemState: true }`, while actual crafting still uses the current item state. This prevents full prefixes/suffixes or occupied affix groups from making the reference list look like PoE2DB data is missing.
- Chaos Orb removal must include desecrated modifiers. This matters especially with Omen of Reduction / lowest-level chaos behavior: a low-level desecrated modifier can be the removal target for Perfect Chaos, and both removal preview highlighting and actual `applyCurrency("chaos")` must use the same candidate set.
- Compatible omens with the same target should stack by merging effects instead of blocking each other. For example, Left Hand Omen of Erasure plus Omen of Reduction should produce a pending chaos omen with `removeType: "prefix"` and `removeLowest: true`; different omen targets remain mutually exclusive.
- Production must not silently rely on fallback sample modifier data. A deployed page showing Perfect Augmentation rolling ring fire resistance `T1 level 80 range 48-52` means `data/poe2db-mod-data.js` failed to load and the app is using fallback data. Runtime now exposes `Core.DATA_STATUS`, the UI warns when PoE2DB data is missing, and smoke asserts perfect augmentation ring fire resistance uses imported PoE2DB `T1 level 82 range 41-45`.
- Tiered currency correctness should be verified with `tools/audit-tiered-currency-matrix.mjs`, not only spot checks. The audit covers all imported bases, all relevant item-level breakpoints, normal/greater/perfect tiers for transmutation, augmentation, alchemy, regal, exalted, and chaos, and asserts every candidate respects currency minimum modifier level, item level, class, base tags, and validation state. As of the 2026-07-10 imported data, Perfect Exalted has `currencyMinModifierLevel: 50`, the full matrix has `failureCount: 0`, and the worst displayed tier number seen for Perfect Exalted is T6; no T7+ Perfect Exalted candidates were found.
- Heavy Belt / `belt_heavy` at item level 82 with imported data has a Perfect Exalted pool of 49 candidates: 17 prefixes, 32 suffixes, minimum modifier level 50. Smoke verifies it can be exalted to 3 prefixes and 3 suffixes; if a deployed page blocks after one prefix, it is stale or not loading imported modifier data.
- Ring prefix pools include percentage elemental damage prefixes from PoE2DB, including `FireDamagePercentage`, `ColdDamagePercentage`, and `LightningDamagePercentage` with T1 level 75 weight 500. The custom-start modifier selects should put rendered modifier text before tier/level metadata so users can find entries like "火焰伤害提高" and "闪电伤害提高" in native select controls.
- Sinister Quarterstaff / 邪恶节杖 is imported as `quarterstaff_sinister_quarterstaff` with class `quarterstaff`. `tools/audit-sinister-quarterstaff.mjs` verifies the user's target craft: locked/fractured T1 crit chance suffix, Chaos Orb can still roll `所有近战技能等级+5` at level 81 weight 100, and suffix Jawbone desecration can still reveal T1 crit multiplier when that group is open.
- `tools/simulate-sinister-quarterstaff-craft.mjs` is a reproducible weighted simulation for the user's Sinister Quarterstaff route. It starts from the stated mid-craft state of three target T1 prefixes, fractured T1 crit chance suffix, and one removable suffix, then uses weighted suffix chaos to hit `所有近战技能等级+5` and weighted suffix Jawbone plus Abyssal Echoes three-choice reveal to hit desecrated T1 crit multiplier.
- Crafting consumption statistics should be derived from `item.history` instead of stored separately. This keeps the displayed cost aligned with undo, custom starts, Abyssal Echoes choices, and locked omen auto-refresh steps.
- Crafting consumption statistics should sit at the top of the main simulator layout, above the crafting panels, rather than inside the lower history panel.
- Modifier appearance chance shown beside a candidate mod should match PoE2DB's equipment-specific prefix/suffix denominator: `mod.weight / total weight of eligible mods with the same type`, not the combined prefix+suffix pool.
- Ordinary jewel bases are imported from PoE2DB `Jewels` `珠宝 物品 /9` as `classId: "jewel"` with `tags: ["jewel", ...]`; they are not tagged as `jewellery`. The old fallback `jewel_abyss` must be hidden whenever real jewel bases are present.
- Ordinary jewel modifiers are imported from PoE2DB `Jewels#JewelMods` with 193 rows: 83 prefixes and 110 suffixes. That table does not expose `DropChance`, so imported jewel rows use equal weight `1` and record this limitation in `raw.weightSource`/source metadata.
- Preserved Cranium on a real jewel base now sees 32 exclusive `JewelsDesecratedMods` rows plus the 193 ordinary jewel base modifiers as desecrated candidates, matching the simulator's existing rule that desecration can include base modifiers.
- Soul Core socketables are imported from PoE2DB ModsView categories into `data/poe2db-crafting-data.js`: Uhtred's Sidereus / chronomancy boots, Kolr's Hunt / marksman gloves, Katla's Gloom / decay gloves, Medved's Tending / soul body armours, Thrudd's Might / destruction weapons, and Vorana's Carnage / berserking helmets. Socketing the Soul Core unlocks a separate `*_modifier` action that rolls the corresponding imported pool.
- Medved's Tending must be modeled as a body-armour Soul Core, not a generic rune. On a current `body_armour_pilgrim_vestments` item level 82 base, the simulator exposes 21 eligible Medved/Soul modifier rows after exact defence-tag matching; smoke verifies it requires socketing, adds one level-65 weight-1 Soul modifier, and rerolls by removing the existing Soul modifier before adding a new one.
- Item base stats that are affected by quality must be displayed through `CraftingCore.baseStatLines(item)`. The raw `base.defenses` rows are the unmodified source data; the UI should not render them directly when showing the current item because armour, evasion, energy shield, rune ward, and weapon physical damage may change with quality.
- Artificer's Orb only opens an empty equipment socket. It should not create stat changes by itself; stat changes come from the later socketed rune or Soul Core effects.
- Vaal Orb state effects now include visible outcomes for rerolling variable modifier values, forcing variable modifiers to their high roll, adding one corrupted extra socket above the normal equipment socket cap, adding one eligible modifier, removing one removable modifier, or no visible change. PoE2DB's cached currency text does not expose branch probabilities, so the simulator must not claim exact Vaal outcome odds without a sourced probability table.
- Ordinary jewel runtime pools come from the individual Ruby, Emerald, Sapphire, Time-Lost Ruby, Time-Lost Emerald, and Time-Lost Sapphire PoE2DB pages. `requiredAnyBaseTags` keeps colour-specific modifiers out of other jewel colours, while Diamond uses the deduplicated union of all three colour pools.
- Jewel rarity caps remain 1 prefix / 1 suffix for magic and 2 prefixes / 2 suffixes for rare, but affix-cap modifiers can expand the effective rare jewel total. `allowed suffix +1` occupies a prefix slot and allows a 2-prefix/3-suffix rare jewel; `allowed prefix +1` occupies a suffix slot and allows a 3-prefix/2-suffix rare jewel.
- Chaos Orb candidate selection on jewels must simulate the exact post-removal state. When a capacity-transfer prefix allows three suffixes, Chaos cannot remove a suffix if that would leave the resulting jewel outside its legal capacity.
- Liquid Emotions are imported from Time-Lost jewel `ModsView.liquid` pools, only apply to rare Time-Lost jewels, remove one legal modifier, and add the matching colour-specific crafted modifier.
- Liquid Emotion replacement is same-side: the removed prefix/suffix determines the type of the newly added crafted modifier. The core candidate preview and execution both enforce this pairing.
- Basic Liquid Emotions must also be imported from PoE2DB `Liquid_Emotions` item cards, not only Time-Lost jewel `ModsView.liquid`. Non-Ancient liquids apply to rare basic Ruby/Emerald/Sapphire/Diamond jewels with the exact card-listed crafted prefix/suffix rows; Ancient liquids remain restricted to Time-Lost jewels. `Liquid_Verisium` is not a basic jewel crafting liquid and should stay out of the liquid emotion action pool.
- The 26 PoE2DB catalysts are imported as actions. Their class restrictions, quality gain, matching-tag value adjustment, and Omen of Catalysing Exaltation quality consumption are covered by core smoke tests.
- Omen of Catalysing Exaltation must affect the actual Exalted roll weights, not only consume catalyst quality. PoE2DB exposes the consumption text but not an exact public multiplier in the cached item page, so the simulator currently models matching catalyst-tag modifiers with `effectiveWeight = baseWeight * (1 + consumedCatalystQuality)` and shows the effective weight/probability in the pool UI. Do not claim this multiplier is a sourced official formula unless a PoE2DB/GGG probability table is later found.
- Sibilant and Skittering catalysts both boost ring `IncreasedCastSpeed` under Omen of Catalysing Exaltation plus right-hand Exalted, but the PoE2DB ring suffix pool makes the resulting chance very low: with 20% quality in the current imported data, the five cast-speed suffix rows are weight 105 out of 87,205, about 0.1204%.
- Essence of the Breach adds a quality-cap modifier. `CraftingCore.qualityCapFor()` should derive the cap from fixed implicits plus current modifiers and apply it to catalyst quality as well as normal item quality. PoE2DB `Quality` lists Breach Ring as `+20% to Maximum Quality` and Refined Breach Ring as `+25% to Maximum Quality`, while Essence of the Breach adds another `+20% to Maximum Quality` for jewellery. Smoke verifies normal jewellery can reach 40%, Breach Ring bases start at 40%, Refined Breach Ring bases start at 45%, and Breach Ring plus Essence reaches 60% before any Vaal infuser overflow.
- Vaal Catalysing Infuser can improve ring/amulet quality while exceeding maximum quality by up to 10%, so Breach Ring plus Essence of the Breach can reach 70% in real-game terms. PoE2DB exposes the +10 overflow text but not the corruption chance; do not model an exact corruption probability unless a sourced table is found.
- Vaal quality overflow currencies are implemented as separate actions: Vaal Armour Infuser, Vaal Whetstone, Vaal Arcanist's Etcher, and Vaal Catalysing Infuser. They require current quality/catalyst quality to be at the current cap, then overflow by up to +10%. Because PoE2DB still does not expose the corruption chance, the simulator marks an unresolved Vaal infuser corruption risk instead of inventing a probability.
- Sovereign Alloy / 君王合金 must use the per-item PoE2DB table instead of card-text inference: weapons add a level-52 suffix, armour/offhand adds a level-20 prefix, and rings/amulets/belts add a level-52 prefix. This prevents the weapon branch from replacing prefixes by mistake.

- Base selection should also avoid one mixed dropdown: top-level base tabs group classes into weapons, armour, offhands, jewellery, jewels, and other before the detailed PoE2DB-backed base select and search are applied.
- Removing a jewel prefix that granted an extra suffix slot must not delete or hide existing suffixes. The UI renders every stored modifier even when the current cap is lower, shows the resulting over-cap state, and smoke covers annulling the transfer modifier from a 2-prefix/3-suffix jewel.
- A jewel that keeps over-cap suffixes after removing an affix-cap prefix must still be able to fill open prefix slots. Existing over-cap mods only block adding more mods to that same over-cap side; they must not consume the opposite side's open slots.
- Explicit prefix and suffix sections should be stacked vertically in the item panel so the layout matches the vertical equipment presentation and remains easy to scan.
